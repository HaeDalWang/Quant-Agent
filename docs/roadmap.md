# 로드맵

각 Phase는 **명확한 완료 기준(Definition of Done)**을 가진다. 기준을 충족하기 전에는
다음 Phase로 넘어가지 않는다. 이것이 "단계적 신뢰" 원칙의 실천이다.

---

## Phase 0 — Foundation (수집·저장·스케줄)

**목표**: 데이터가 매일 자동으로, 안정적으로 쌓인다.

### 작업
- [ ] `collectors/base.py` — Collector 인터페이스
- [ ] `collectors/price.py` — FinanceDataReader 어댑터 (KR+US 시세)
- [ ] `storage/base.py` + `storage/duckdb_store.py` — 멱등 저장
- [ ] `config/` — 종목 유니버스(Tier 1), `.env` 시크릿 로딩 + 검증
- [ ] `scheduler/` — 장마감 후 일별 수집 잡 (시장별 분리)
- [ ] 종목 단위 장애 격리 + 재시도/백오프
- [ ] 구조화 로깅

### 완료 기준 (DoD)
- ✅ Tier 1 종목의 일별 OHLCV가 DuckDB에 자동 적재된다.
- ✅ 잡을 두 번 돌려도 중복 없이 멱등하게 저장된다.
- ✅ 한 종목 수집 실패가 전체를 죽이지 않는다.
- ✅ 시크릿 누락 시 시작 단계에서 명확히 실패한다.

> **AI 없음.** 순수 시스템.

---

## Phase 1 — Intelligence (분석·알림·리포트)

**목표**: 데이터에서 신호를 뽑고, 조건 충족 시 Discord로 알림이 오고, 정기 리포트가 자동 생성된다.

### 작업
- [ ] `analysis/indicators.py` — MA/RSI/MACD/ATR (pandas-ta 래핑, 순수 함수)
- [ ] `analysis/signals.py` — 지표 → 신호 변환
- [ ] `alerts/base.py` + `alerts/discord.py` — AlertChannel + Discord webhook
- [ ] `alerts/rules.py` — 규칙 엔진 (조건 → 트리거)
- [ ] `universe/screener.py` — Tier 2 동적 스캔
- [ ] `reports/` — 정기(일/주간) 마크다운 리포트
- [ ] DART 공시·뉴스 수집 추가 (`collectors/disclosure.py`, `news.py`)

### 완료 기준 (DoD)
- ✅ 정의된 규칙(예: RSI 과매도 + 거래량 급증) 충족 시 Discord 알림이 온다.
- ✅ 매일/매주 종합 리포트가 자동 생성·전송된다.
- ✅ Tier 2 스크리너가 조건 기반으로 후보를 편입/이탈시킨다.
- ✅ 모든 신호·선정에 구조화된 사유 로그가 남는다.

> **AI 없음.** 규칙 기반. 이 기반이 단단해야 위에 에이전트를 신뢰할 수 있다.

---

## Phase 2 — Agent (단일 에이전트, 관찰·판단·제안)

**목표**: LLM이 Phase 0~1 산출물을 컨텍스트로 받아 "관찰·판단·제안"한다. **실거래 없음 (paper).**

### 작업
- [ ] `agents/` — LangGraph 기반 단일 에이전트
- [ ] 컨텍스트 빌더: 저장소 데이터 → 구조화 프롬프트 (채널이 아니라 Store에서 읽음)
- [ ] paper trading 장부 (모의 포지션·손익 추적)
- [ ] 제안 출력: 근거 포함된 매수/매도/보유 의견 → Discord
- [ ] 의사결정 로그 (재현·감사 가능)

### 완료 기준 (DoD)
- ✅ 에이전트가 매일 유니버스에 대해 근거 있는 제안을 생성한다.
- ✅ 모든 제안이 모의 장부에 기록되고 사후 성과가 추적된다.
- ✅ 제안의 입력 컨텍스트가 재현 가능하다.

> ⚠️ 절대 실거래 금지. 신뢰를 "증명"하는 단계.

---

## Phase 3 — Agent Group (멀티에이전트 토론·합의)

**목표**: 역할 분담 에이전트들이 토론하고 합의안을 만든다.

### 작업
- [ ] 역할 에이전트: 펀더멘털 / 기술적 / 리스크 / 포트폴리오 매니저(PM)
- [ ] 토론 그래프 (LangGraph) — 각 역할이 의견 제시, PM이 종합
- [ ] Discord 멀티 페르소나 (웹훅별 아바타) + 종목별 스레드
- [ ] 합의 → 근거 포함 제안 (여전히 paper)
- [ ] 역할 간 이견·리스크 플래그 가시화

### 완료 기준 (DoD)
- ✅ 각 역할이 독립적 관점으로 의견을 내고, PM이 합의안을 도출한다.
- ✅ Discord에서 토론 과정이 종목별 스레드로 추적된다.
- ✅ 모의 성과가 Phase 2 단일 에이전트 대비 평가된다.

---

## Phase 4 — Execution (자율 실행, 점진 도입)

**목표**: 사람 승인 게이트를 거쳐 소액 실매매. 신뢰가 쌓이면 점진적으로 자동화.

### 작업
- [ ] 브로커 추상화 (`broker/base.py`) + 첫 구현체 (KIS or Alpaca)
- [ ] `AlertChannel.request_approval` — Discord 버튼 승인 게이트
- [ ] 리스크 한도: 종목당/일일 최대 손실, 포지션 상한, 킬 스위치
- [ ] 단계적 자율화: 전건 승인 → 한도 내 자동 → 확대
- [ ] 실시간 모니터링·비상 정지

### 완료 기준 (DoD)
- ✅ 모든 주문이 승인 게이트를 거친다 (초기).
- ✅ 리스크 한도가 코드로 강제된다.
- ✅ 킬 스위치가 즉시 동작한다.
- ✅ 소액 실거래 성과가 모의 대비 추적·검증된다.

> ⚠️ **고위험 단계.** 소액부터. 사람이 항상 멈출 수 있어야 한다. 자율화 확대는 검증된 신뢰에 비례해서만.

---

## 횡단 원칙 (모든 Phase 공통)

- **재현성**: AI 경계 아래는 결정론적·멱등.
- **시크릿**: `.env`만, 커밋 금지, 시작 시 검증.
- **테스트**: 신규 기능은 테스트 동반 (수집 어댑터는 모킹, 지표는 단위 테스트).
- **불변 데이터**: 수집·분석 입출력은 새 객체 반환.
- **작은 모듈**: 파일당 200~400줄 지향, 800줄 상한.
